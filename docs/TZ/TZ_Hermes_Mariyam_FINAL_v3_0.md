### 0.6. Изменения v3.5 → v3.6 (решение заказчика 2026-07-12) — детерминированная identity binding

Инцидент: при livete-тестировании бухгалтерских tools Telegram-сессия test-user была корректной, но Hermes вызвал tools с `user_id` администратора — две тестовые записи попали владельцу admin. Root cause: стабильный Telegram sender ID отсутствовал в model context, поэтому identity ошибочно зависела от LLM. Решение (зафиксировано в ТЗ, см. §15, §19, §21; детали — `docs/TZ/EVIDENCE_IDENTITY_GUARD_2026-07-12.md`):

1. `user_id` для user-scoped MCP tools **нельзя** доверять LLM, display name или аргументу модели.
2. Identity определяется **внутри Hermes** через официальный profile plugin и `tool_execution` middleware: current session → persisted Telegram sender → приватный mapping → internal `users.id`.
3. `role=oyijon` всегда работает только со своим internal `user_id` (self-only, regardless of requested).
4. `role=admin` может обращаться к другому target только если tool входит в строгий allowlist (`get_expense_report`, `get_balance_summary`, `get_admin_report_data`, `save_plan_note`) **и** target входит в `allowed_target_user_ids`.
5. Admin cross-target write/delete запрещены (только указанные read/plan-note tools).
6. Unknown session/sender, malformed mapping, неизвестная роль и небезопасные права файла блокируются **fail-closed до MCP tool** (коды `IDENTITY_UNRESOLVED` / `IDENTITY_TARGET_FORBIDDEN` / `IDENTITY_MAPPING_INVALID` / `IDENTITY_MAPPING_PERMISSIONS` / `IDENTITY_GUARD_ERROR`).
7. Mapping хранится **вне git** и вне model-visible profile, права POSIX `0600` (plugin отказывает при более широких правах).
8. Backend, Gateway core, DB schema и tool contracts API **не становятся вторым мозгом** и **не выполняют identity routing** — они остаются tools/storage (см. §20).

**Статус runtime identity (2026-07-14):** plugin `mariyam_identity_guard` **1.0.4** на VPS; Stage 5 и Stage 5.1 E2E PASS (см. `EVIDENCE_STAGE_5_E2E_2026-07-12.md` и `../EVIDENCE_STAGE_5_1_LIVE_2026-07-13.md`). Не меняет архитектуру Hermes-first.

### 0.7. Изменения v3.6 → v3.7 (решение заказчика 2026-07-13) — аналитика расходов и месячный план

**Зачем:** Мариям — не калькулятор, а бытовой финансовый аналитик: группы расходов, товары, количество, сравнение месяцев, тренд, план/факт и осторожные советы (опт только при фактических ценах).

1. **Расширение `transactions` (nullable, старые rows валидны):** `item_name_normalized`, `quantity`, `unit` (canonical: `kg|g|l|ml|pcs|pack`). Quantity > 0; unit только вместе с quantity; если количество не сказано — NULL (не угадывать).
2. **Нормализацию товара/quantity/unit делает Hermes**; backend только хранит и агрегирует.
3. **`get_expense_report`:** optional `compare_previous`, `trend_months` (default 3, max 12); выходы: `by_item`, `previous_period`, `monthly_series`; `change_percent=null` если прошлый total=0.
4. **План бюджета:** таблица `monthly_budget_plans` + tools `set_monthly_budget`, `get_monthly_budget_status` (plan vs fact).
5. **Счёт / проза:** backend считает факты (суммы, counts, compare, plan/fact); Hermes объясняет и даёт осторожные советы. LLM memory **не** источник финансовой аналитики.
6. **Запрет:** выдумывать quantity/цены/экономию; утверждать «опт дешевле» без фактических цен; точные прогнозы без данных.
7. **Tools count (требование):** после реализации Этапа 5.1 — 21. Identity guard обязателен для всех новых user-scoped tools.
8. Требования Этапа 5.1 не меняются; offline milestone зафиксирован в §0.8, live acceptance — в §0.9. Реальная Ойижон до handover не подключается.

## 1. Главная идея: Hermes-first

Hermes Agent — это готовый самоулучшающийся агент: у него уже есть Telegram/Discord/Slack gateway, изолированные профили (своя память, сессии, skills), система skills, встроенный cron-планировщик с доставкой в любой канал, поддержка MCP-инструментов и авто-транскрипция голоса (STT) + синтез речи (TTS). **Мы не пишем свой «мозг» — мы конфигурируем Hermes и даём ему точные инструменты.**

```text
Hermes Agent  = мозг: понимание речи, стиль Мариям, память, рассуждение,
                выбор когда вызвать tool, голос, Telegram, cron, обучение Ойижон.
Backend/tools = руки и память фактов: сохранить/прочитать расход, отчёт,
                прогресс Корана, health note, alert, backup. Никакой «логики смысла».
```

### 1.1. Что делает Hermes (и только Hermes)

- Всё общение с Ойижон и Бахриддин ака.
- Понимание бытовых фраз (текст и голос), в т.ч. что это «расход», «напоминание», «жалоба».
- Классификацию расхода по категории (из фиксированного списка) — **до** вызова tool.
- Нормализацию сумм («180 минг» → 180000) **до** вызова tool.
- Формирование текста всех сообщений и отчётов (Ойижон и админу).
- Мягкую память о людях, привычках, предпочтениях.
- Планирование по cron (утро, вечер, 19:30, новости).
- Веб-поиск для новостей и вопросов Ойижон.

### 1.2. Что делает backend/tools (и только это)

- Принимает **уже разобранные** данные от Hermes и сохраняет их.
- Возвращает **точные числа и факты** (итоги, отчётные данные) по запросу Hermes.
- Валидирует входные данные (например: категория есть в списке; сумма — целое ≥ 0).
- Делает backup, отдаёт статус системы, логирует стоимость API.

Backend **не** анализирует чат, **не** угадывает намерение, **не** пишет прозу отчётов, **не** решает «когда» что-то отправить.

## 5. Первый запуск и onboarding

### 5.1. Механизм (важно для исполнителя)

Onboarding управляется **флагом прогресса в Hermes memory + cron**, не жёстким календарём.

- В memory профиля хранится `onboarding_step` (0..5) и `onboarding_last_shown_date`.

## 6. Голос и модели (STT / TTS / LLM)

### 6.1. Поток голоса

Voice от Ойижон → Gateway получает файл → STT → Hermes понимает → при необходимости вызывает tool → Мариям отвечает **текстом** (узбекская кириллица).

**Решение заказчика (v3.2): TTS отключён — все ответы Мариям только текстом.** Голосовые сообщения от Ойижон принимаются и распознаются как обычно. При включении TTS в будущем вернётся правило экономии: длинное — текстом, голосом только короткая суть.

### 6.2. STT — выбор после теста на реальном голосе (Этап 0/3)

STT заранее не фиксируется. На тест-сете из §15 (Этап 0) сравнить кандидатов:

- ElevenLabs Scribe (для узбекского показывает лучшую точность из проверенных: ~3.1% WER FLEURS / ~5.5% Common Voice);
- Hermes native STT (Whisper по умолчанию) — для узбекского обычно слабее, особенно на числах и диалекте;
- OpenAI/Groq Whisper API;
- local Whisper как офлайн-fallback.

Проверяем: понимание узбекского и **хорезмского** произношения; **числа** («минг», «миллион») без потерь; латиница/кириллица; скорость; стоимость. **Рекомендация ревизора:** по умолчанию брать ElevenLabs Scribe, если тест это подтвердит.

**Уточнение v3.2:** STT не обязан выдавать чистую кириллицу — сырой транскрипт (латиница, с ошибками) интерпретирует LLM (Hermes) и отвечает кириллицей. Критерий выбора STT — **сквозная точность чисел** (голос → сохранённая сумма в БД), а не WER транскрипта. Текущий рабочий вариант — OpenAI Whisper; если сквозная точность на тест-сете <90% — сравнить кандидатов из списка выше (включая Gemini-аудио через OpenRouter).

### 6.3. TTS

**Отложено (v3.2):** TTS отключён решением заказчика 2026-07-10 — раздел применяется только при будущем включении голосовых ответов.

Женский, мягкий, спокойный, понятный, не роботизированный, приемлемый по цене. Кандидаты: ElevenLabs / OpenAI TTS (Opus нативно, удобно для Telegram) / Edge TTS (бесплатно, но узбекский слабее и нужен ffmpeg для Opus) / Hermes-supported TTS. Выбрать после короткого прослушивания на узбекской кириллице.

### 6.4. LLM

Одна основная сильная мультиязычная модель с хорошим узбекским для всего общения; **эскалацию на более сильную делать редко** (только сложный анализ), чтобы не переусложнять и не дорожить. Модель фиксируется в профиле (`hermes model`). Не строить сложный per-message роутер. Модель выбрать после мини-теста на узбекских фразах.

### 6.5. Гарантия кириллицы

(1) Жёсткое правило в skill; (2) пост-валидация выхода: если найдены латинские буквы в узбекском тексте — транслитерация в кириллицу перед отправкой; (3) автотест (см. 3.1).

### 6.6. Бюджет API

Цель: **≤ 10–15 USD/мес.** До запуска — прикидка: кол-во voice/день, средняя длина, кол-во TTS-ответов, LLM-запросов, цена STT/TTS/LLM. Если прогноз выше — сократить voice-output, взять дешевле модель, кэшировать новости/погоду/намаз. Факт логировать в `usage_costs` (см. 11).

## 7. Бухгалтерия

### 7.1. Ввод без кнопок

Ойижон говорит естественно: `Бугун бозордан гўшт 180 минг, нон 12 минг, сабзи 20 минг олдим.`
Hermes: понимает, что это расход → нормализует суммы → (если сказано) quantity/unit → item_name_normalized → определяет категории из фиксированного списка → вызывает `save_expense` → отвечает:

```text
Ойижон, ёзиб қўйдим: гўшт 180 000 сўм, нон 12 000 сўм, сабзи 20 000 сўм. Жами 212 000 сўм.
```

Если сказано количество (`10 кг картошка`): Hermes передаёт `quantity`/`unit`/`item_name_normalized`. Если не сказано — quantity/unit = NULL, **не угадывать**.

### 7.2. Подтверждение при сомнении (критично для голоса)

Если сумма/товар распознаны неуверенно — **не сохранять**, переспросить:

```text
Ойижон, гўшт 180 минг сўм дедингизми? Тўғри тушундимми?
```

Подтвердила → сохранить. Нет → уточнить. **Молча неверную сумму не сохранять никогда.**

### 7.3. Исправление и удаление (новое, обязательно)

Ойижон может ошибиться и поправить:

```text
«Йўқ, гўшт 150 минг эди»      → Hermes зовёт update_last_expense / update_expense
«Охиргисини ўчириб ташла»      → Hermes зовёт delete_last_expense
«Кечаги нонни ўчир»            → Hermes зовёт delete_expense по найденной записи
```

Мариям подтверждает изменение простыми словами.

### 7.6. Аналитика и месячный план (v3.7 / Этап 5.1)

- объясняет отчёт;
- осторожные советы: «часто покупалось», «выросло vs прошлый месяц», «можно сравнить цену большой упаковки», «сухие продукты удобнее реже и больше» — **только** с проверкой цены, срока, хранения, бюджета.

**Запрет советов:** выдуманная экономия %; «точно дешевле оптом» без фактов; quantity=1 без данных; смешение units.

**Статус v3.9:** **CLOSED / LIVE PASS**. Repo и VPS runtime = 21 tools / plugin 1.0.4; migration 002, canonical SKILL и skill-protect активны; controlled E2E подтверждён.

## 8. Напоминания, планы, утро/вечер

### 8.1. Механизм = Hermes cron (backend-scheduler НЕ строить)

Все напоминания и периодические сообщения — через **встроенный cron Hermes** с доставкой в Telegram. Отдельный планировщик-сервис не создаётся. Напоминания хранятся в Hermes (профиль), поэтому попадают в backup через копию профиля (см. 13). Тексты-заметки/счётчики, если нужно сохранить как факт, идут в `save_plan_note`.

### 8.2. Создание из речи

```text
Ойижон: «Эртага соат 10 да дорини эслат.»
Мариям: «Хўп, Ойижон. Эртага соат 10:00 да дорингизни эслатаман.»
Бахриддин ака: «Завтра в 10:00 напомни маме выпить лекарство.» → Hermes создаёт cron-напоминание для Ойижон.
```

Типы: лекарства (по запросу), Коран, планы дня, бытовые дела, звонки родным, намаз, custom.

### 8.3. Лекарства в MVP

Строгого назначенного списка нет → **отдельный medication-модуль не делаем.** Просьбу напомнить лекарство сохраняем как обычное cron-напоминание. Точную таблицу лекарств добавим позже, если появятся назначения.

### 8.4. Утреннее и вечернее сообщение

- Утро (не слишком рано): салом, погода, короткий план, мягкая фраза.
- Вечер: если за день не было данных о расходах/Коране/состоянии — мягко спросить (без повторов, если молчит):
```text
Тинчликми, Ойижон? Бугун харажат бўлдими? Қуръон ўқидингизми? Дориларни ичдингизми?
```

### 9.1. Новости (ежедневно)

3–5 коротких новостей, простым узбекским кириллицей, без тревоги и перегруза, из доверенных источников (kun.uz, gazeta.uz, daryo.uz; официальные — для гос/соц тем; мировые топ — для интересных международных). Реализуется через веб-поиск Hermes + Hermes формулирует. Кэш на день.

```text
Ўзингизни қизиқтирган саволлар бўлса, бемалол айтаверинг, мен интернетда ишончли манбалардан тезда қидириб топиб, сизга тушунтираман.
```

## 10. Медицинская безопасность

### 10.1. Роль

Мариям — не врач. **Нельзя:** ставить диагноз, назначать/менять/отменять лекарства, спорить с врачом, обещать «ничего страшного». **Можно:** мягко поддержать, предложить сказать близким и обратиться за медпомощью, записать `health_note`, уведомить Бахриддин ака.

### 10.2. Срочные фразы — двойное срабатывание (надёжность)

Триггеры (список расширяемый): `юрагим оғрияпти`, `кўкрагим оғрияпти`, `нафас олиш қийин`, `бошим айланяпти`, `ёмон бўляпман`, `ҳушим кетяпти`, `қон босимим жуда баланд`.

Детекция делается **двумя способами одновременно** (belt & suspenders):

1. **LLM** (Hermes-skill) — понимает смысл и перефразировки.
2. **Детерминированный keyword-предохранитель** — простой список корней/фраз поверх LLM. Если сработал **любой** из двух → эскалация.

Это не новый канал помощи (решение по 103 и второму контакту см. 10.3) — это гарантия, что **уже согласованное** уведомление админа не будет пропущено из-за ошибки модели/транскрипции.

**Где живёт детекция (уточнение v3.1):**

- Оба уровня детекции живут **в Hermes** (skill / конфигурация профиля). Backend сообщений не видит и детекцию не делает — переносить её на backend запрещено (§20).
- Если установленная версия Hermes поддерживает детерминированные pre-processing hooks на входящие сообщения — keyword-предохранитель реализуется **там** (истинно детерминированно).
- Если hooks нет — предохранитель реализуется жёсткой инструкцией в skill. Это компромисс (инструкцию исполняет LLM), поэтому гарантия обеспечивается acceptance-критерием Этапа 7: **recall уведомления админа = 100% на обязательном тест-сете**. Без пройденного тест-сета Этап 7 не сдаётся.

**Правила списка keyword (v3.1):**

- Список — **узкий**: только корни конкретных срочных фраз из перечня выше (`юрагим оғри`, `кўкрагим оғри`, `нафас ол`, `бошим айлан`, `ҳушим кет`, `қон босим`, `ёмон бўляпман`).
- **Запрещено** включать одиночные общеупотребительные слова (`ёмон`, `ёрдам`, `дард`, `бемор`, `температура` без уточняющего контекста) — они дают ложные эскалации и спам админу, из-за чего админ перестанет читать алерты.
- Расширять список только по фактам пропусков на реальных фразах, не «на всякий случай».

Действия при срабатывании:
1. Мягкий серьёзный ответ Ойижон.
2. Совет сказать близким и обратиться за медпомощью.
3. Уведомить Бахриддин ака.
4. Записать `alert_event`.

```text
Ойижон: Ойижон, бу жиддий бўлиши мумкин. Илтимос, яқинларингизга айтинг ва тиббий ёрдамга мурожаат қилинг. Мен Ўғлингизга ҳам хабар бераман.
Админу:  Бахриддин ака, муҳим хабар: Ойижон “юрагим оғрияпти” деди. Мен унга яқинларига айтишни ва тиббий ёрдамга мурожаат қилишни айтдим.
```

### 10.3. Решения заказчика (зафиксировано)

В MVP: **второй экстренный контакт не добавляется**; **автоматический жёсткий совет «103 га қўнғироқ қилинг» не включается** (Мариям мягко советует медпомощь). Причина: Hermes не первая и не последняя линия помощи. Можно включить позже настройкой.

### 13.2. Миграции и seed (обязательно, v3.1)

- SQL-файлы нумеруются: `001_init.sql`, `002_*.sql`, … Каждый файл **идемпотентен** (`CREATE TABLE IF NOT EXISTS`, `ON CONFLICT DO NOTHING`).
- **Текущий runtime:** migration 001 + 002 active. **План v3.12:** migration 003 (`monthly_budget_items` с price snapshot, `monthly_plan_cycles`), 004 (`utility_accounts`, `utility_snapshots`) и 005 (`recurring_obligations`) — **PLANNED / NOT IMPLEMENTED**; файлов и production schema для них ещё нет.
- Для utility snapshots правило отрицательного prepaid balance против отдельного debt утверждается только после исследования реального кабинета; без явного provider rule одновременно хранить оба представления долга запрещено.
- `docker-entrypoint-initdb.d` применяет скрипты **только при первом создании volume**. Любое изменение схемы после этого применяется вручную: `docker compose exec hermes_mariyam_postgres psql -U hermes -d hermes -f /docker-entrypoint-initdb.d/00N_*.sql`. Эта команда документируется в deploy-доке.
- **Seed пользователей (v3.4):** обязателен `role=admin` (Бахриддин ака) — до первого реального вызова tools в таблице `users` он должен существовать. Опционально разрешён **временный test-user** для end-to-end тестов: `role=oyijon`, `display_name="Тест Ойижон"`, **только на втором Telegram-аккаунте, контролируемом заказчиком** (не реальная Ойижон). Настоящий ID Ойижон и настоящий seed `role=oyijon` выполняются **только при handover** (§0.4, §21). Создание — через tool `ensure_user` (см. 15.15) при настройке профиля Hermes, либо документированным SQL-скриптом в deploy-доке. Перед handover временный test-user и его данные удаляются. Первый вызов `save_expense` на пустой таблице `users` (без admin) не должен быть возможен «по недосмотру».

## 14. Время и локаль (обязательное правило)

- Хранение всех меток времени: **UTC**, тип `TIMESTAMPTZ`.
- «Сегодня» / «бугун», границы дня/недели/месяца, ежедневный отчёт 19:30, утро/вечер — считаются в **Asia/Tashkent (UTC+5)**.
- Пример: «бугунги харажат» = транзакции с `occurred_at` в диапазоне [сегодня 00:00 Ташкент, завтра 00:00 Ташкент), переведённом в UTC.
- Все cron-расписания Hermes задаются в Asia/Tashkent.

Это правило предотвращает классический баг сдвига на 5 часов.

## 15. Контракты инструментов (Tools I/O) — приложение для исполнителя

Все tools — часть одного MCP-сервера (см. раздел 16). Единые конвенции:

- Успех: `{ "ok": true, ... }`
- Ошибка: `{ "ok": false, "error_code": "STRING", "message_ru": "...", "message_uz": "..." }`
- Суммы: `amount_uzs` / `amount` — целое или NUMERIC в **сумах** (тийины не используем). Нормализацию из речи делает Hermes.
- Время: входные/выходные метки — **UTC ISO 8601** (`2026-07-09T14:30:00Z`). `occurred_at` может прийти как дата — backend трактует день в Asia/Tashkent.
- `user_id` — внутренний id из `users`. Маппинг telegram_id → user_id создаётся/находится через `ensure_user` (15.15); полученные user_id фиксируются в памяти профиля Hermes при настройке.
- **Детерминированная identity binding (v3.6):** для user-scoped tools effective `user_id` определяется **внутри Hermes** через profile plugin `mariyam_identity_guard` (`tool_execution` middleware), а **не** значением из аргументов модели, display name или ответа LLM. Поток: current Telegram session → persisted sender → приватный mapping (вне git/profile) → internal `users.id`. `role=oyijon` всегда принудительно получает свой собственный `user_id` (self-only). `role=admin` может обращаться к чужому target только если tool входит в строгий allowlist (`get_expense_report`, `get_balance_summary`, `get_admin_report_data`, `save_plan_note`) **и** target входит в `allowed_target_user_ids` из mapping; cross-target write/delete запрещены. Unknown session/sender, malformed mapping, неизвестная роль и небезопасные права mapping-файла блокируются **fail-closed до MCP tool** (backend не видит вызова). Backend API tools не меняется (см. §20, §19).
- **Per-tool схемы (v3.1):** каждый tool публикует **собственную** MCP inputSchema с перечислением своих полей и блоком `required`. Одна общая схема на все tools запрещена — она ухудшает точность tool-calling и валидацию.
- **Честные заглушки (v3.1):** не реализованная функция возвращает `{ "ok": false, "error_code": "NOT_CONFIGURED", ... }`. Возвращать «успех» или «всё хорошо» из заглушки запрещено.

**Коды ошибок (единый список, v3.1):**

| Код | Когда |
|---|---|
| `BAD_CATEGORY` | категория не из фиксированного списка |
| `BAD_AMOUNT` | сумма отрицательная / не число |
| `INVALID_INPUT` | не хватает обязательных полей, неверный формат даты/валюты/severity, пустые `fields` в update |
| `NOT_FOUND` | запись для update/delete/get не найдена |
| `NOT_CONFIGURED` | функция ещё не настроена (backup до Этапа 8) |
| `UNKNOWN_TOOL` | неизвестное имя tool |
| `INTERNAL` | прочая внутренняя ошибка (в норме не возникает) |

Валидируемые enum-поля (`currency`, `severity`, `source_type`, `detected_by`, `service_type`) проверяются **в коде до INSERT** и дают `INVALID_INPUT` с понятным сообщением, а не ошибку CHECK-констрейнта из БД.

## 16. Подключение tools к Hermes (MCP)

Hermes подключает внешние инструменты через **MCP (Model Context Protocol)** — транспорт **stdio** или **HTTP**.

1. Backend реализуется как **MCP-сервер**, экспортирующий tools из раздела 15 (имена, входные/выходные схемы — как описано).
2. Регистрация в профиле `mariyam_oyijon`: добавить MCP-сервер в конфигурацию Hermes (через `hermes` CLI / конфиг профиля), задать транспорт и адрес/команду запуска. Проверить, что `hermes tools` показывает подключённые tools.
3. Выдать tool-permissions: разрешить агенту вызывать нужные tools.
4. Секреты (DB URL, API-ключи) — только в `.env` / переменных окружения, не в коде и не в конфиге под git.
5. **Транспорт (решение v3.1): по умолчанию на VPS — stdio.** Hermes запускает backend как subprocess; docker compose при этом используется только для PostgreSQL. Это убирает HTTP-слой целиком (меньше движущихся частей).
6. HTTP-транспорт — запасной вариант (если stdio с установленной версией Hermes неудобен): слушать **только 127.0.0.1**, использовать поддерживаемый session-manager из официального MCP SDK (например `StreamableHTTPSessionManager`), а не самодельную обвязку на приватных API. Работоспособность подтверждается реальным MCP-клиентом (initialize + tools/list + tools/call) до деплоя.
7. **Пул соединений (v3.1):** backend держит **один** пул asyncpg на процесс, создаваемый один раз при старте (или лениво при первом вызове с кэшированием). Создание пула на каждый tool-вызов запрещено — это утечка соединений, которая кладёт Postgres.

Ссылки на актуальную документацию Hermes по MCP и tools — в разделе «Источники». Точные имена CLI-команд/полей конфигурации исполнитель сверяет с документацией версии Hermes, установленной на VPS (`hermes --version`).

Долгоживущие процессы (например Hermes, если не в docker) — обычный `Type=simple` + `Restart=always`.
- Юнит проверяется командой `systemd-analyze verify <unit>` до включения.

**Статус расширения обязательных платежей: PLANNED / NOT IMPLEMENTED.** Текущие требования Stage 6 к Hermes cron сохраняются.

### Этап 7 — Админ-отчёты и safety

## 23. Риски и меры

- **STT узбекского/чисел/хорезмского** → тест на реальном голосе; переспрос сумм; нормализация числительных в Hermes; fallback на текст.
- **Стоимость cloud API** → короткие voice, длинное текстом, дешёвая модель для простого, кэш новостей/погоды/намаза, `usage_costs`.
- **Медицина** → не диагноз; совет близких/медпомощь; двойная детекция алертов; уведомление админа.
- **Кириллица** → правило skill + пост-валидация/транслитерация + автотест.
- **Смешение версий ТЗ** → этот документ единственный; v1/v2.0/v2.1 в архив; кодеру отдавать только этот файл.
- **Молчаливый простой бота** → автозапуск, heartbeat, внешний uptime-monitor.
- **Тихая деградация backend** (утечка соединений/ресурсов, лгущие заглушки) → один пул на процесс, smoke-тест на ≥50 вызовов, честные `NOT_CONFIGURED`, heartbeat через `get_bot_status`.
